SYSTEM ARCHITECTURE AND REQUEST LIFECYCLE
------------------------------------------
Every request to EDITH follows this pipeline: Input arrives via one of three entry points — the FastAPI web server on port 8001 (chat_server.py), the Telegram bot (telegram_bot.py), or the always-on voice wake listener (wake_listener.py). The web server is a thin shell; all routes live in the routes/ directory. After input arrives, session.py starts a session with device tracking and UUID. The input goes to intent.detect_intent() which runs 30+ regex pattern groups in a priority-ordered cascade, then falls through to a sklearn LogisticRegression ML model (60 training samples, TF-IDF vectorizer), and finally to a Groq LLM semantic fallback if both fail. Intent classification returns a string constant like "chat", "email", "code", "agent", "council", etc. The orchestrator then runs recall() to pull relevant memories from the 4-tier memory system. It calls detect_emotion_urgency() from ml_router.py to get emotional context (frustrated/happy/confused/stressed/neutral) and urgency (LOW/MEDIUM/HIGH). The conversation DNA engine (conversation_dna.py) analyzes time of day, device type, emotion, session length, and recent intents to generate a style instruction injected into the system prompt. The full system prompt for EDITH identifies her as Vaibhav's personal AI, tells her to be warm but not sycophantic, to never say "Great question" or "Certainly," to match his energy, to be honest even when uncomfortable, and to never break character. The prompt explicitly mentions that Vaibhav is a solo developer, runs Manjaro Linux on CPU-only hardware, has a zero-cost philosophy, and is technically sharp. After the system prompt is built, smart_router.py routes the request to the appropriate cloud provider. The response is returned, memories are written asynchronously, knowledge graph triples are extracted in a background thread, cognitive profile is updated, and the session is tracked.

The detection function runs them in a specific priority order to prevent collisions — for example, calendar_create must be checked before code patterns because "create an event" would match CODING_PATTERNS. The final LLM fallback calls smart_call with a prompt asking it to classify into one of nine categories. The system has documented collision problems between calendar and vision patterns, and between code and agent patterns.


LLM ROUTING — SMART ROUTER
---------------------------
core/smart_router.py is the brain of EDITH's AI routing. It manages a 4-provider cloud chain: Groq (llama-3.3-70b-versatile, 150 calls/day limit), Gemini (gemini-2.0-flash with Google Search grounding enabled, 250/day), NVIDIA NIM (meta/llama-3.1-70b-instruct, 80/day), and OpenRouter (meta-llama/llama-3.3-70b-instruct:free, 80/day). All providers are called with the same EDITH persona prefix that instructs the LLM to respond as EDITH — Vaibhav's personal AI, sharp and warm, never sycophantic. Each provider has a circuit breaker (CLOSED/OPEN/HALF_OPEN states), exponential backoff starting at 60 seconds with max 5 minutes, and per-provider daily call counters stored in SQLite. The router has a response cache using LRU with 100 entries and 1 hour TTL, with context-aware fingerprinting that includes a time bucket (hourly) and top entity so cached answers become stale correctly. There is latency tracking per provider using a rolling window of 20 samples, and a latency leaderboard that auto-prefers the fastest available provider. A tuner system (utils/tuner.py) reads weekly feedback data from thumbs up/down traces and adjusts provider weights by ±10% per week maximum, persisted in tuner_state.json. The routing chain order is also configurable per task type via environment variables (PROVIDER_ORDER_CODING, PROVIDER_ORDER_REASONING, etc.). Privacy routing is enforced: intents classified as "vault", "shell", "email", "phone", "call", "sms", "rag", and "create_file" are forced to local Ollama only. On the cloud node (EDITH_NODE_TYPE=cloud), these private intents are rejected entirely. PII detection (email addresses, phone numbers, SSN, credit cards, Aadhaar, credentials) forces local-only routing. Chain-of-thought injection (CoT) is always appended to system prompts. A FORCE_DEEP_THINK flag (default True) adds step-by-step reasoning instructions. The router also supports streaming via smart_stream() which yields tokens synchronously for SSE response streaming in the web UI.

MEMORY SYSTEM — FOUR TIERS
---------------------------
EDITH has four tiers of memory all wired together in orchestrator.recall(). Tier 1 is the hot RAM cache in smart_memory.SmartMemoryManager — an OrderedDict capped at 50 items with LRU eviction. Tier 2 is warm SQLite (memory_archive.db) with full-text search via FTS5 virtual tables, indexed by category and timestamp. Recency decay scoring uses exponential decay with a 14-day half-life so older memories rank lower. Tier 3 is semantic ChromaDB vector store (memory_db/) using HuggingFace all-MiniLM-L6-v2 embeddings via LlamaIndex. Notes in the notes/ directory are indexed here, including about_me.txt (currently only 5 lines). Tier 4 is the NetworkX knowledge graph (memory_db/edith_graph.json) which stores entity-relation-entity triples. The graph is queried by fuzzy node matching with configurable traversal depth. After every conversation turn, _post_turn_reflection() fires asynchronously: it asks the LLM "is there a fact worth saving here?" and if so, stores it and also calls extract_and_store_triples(). The background daemon runs extract_daily_graph_triples() at noon daily using a regex-based (not LLM-based) triple extractor on recent queries. Episodic memory (episodic_memory.py) saves entire session conversation transcripts as searchable episodes in a separate ChromaDB collection, queryable by semantic similarity. The context compression system (compress_context() in smart_memory.py) deduplicates overlapping RAG chunks before they hit the LLM using SequenceMatcher similarity at a 0.7 threshold. Memory recall in orchestrator.chat() fetches from Tier 1+2, adds graph facts, falls back to ChromaDB, and only pulls 3 items total into the context. The about_me.txt file has only 5 lines of personal context: name is Vaibhav, working on AyurStock Pro, lives in Fatehpur UP India, building EDITH, hardware specs.


VOICE PIPELINE
--------------
voice.py implements the full voice stack. Wake word detection runs in wake_listener.py using either openwakeword (neural, preferred, responds to "Hey EDITH") or Vosk (offline, fuzzy match fallback). Vosk uses a small English model stored in models/vosk/. The wake listener continuously reads 4000-frame audio chunks, resampling if the hardware device doesn't support 16kHz natively, tries up to 4 different device/rate combinations. When wake word is detected, it fires _trigger_wake_sequence() which speaks a greeting via the local bridge HTTP server, then listens for user input via Vosk transcription with a 30-second timeout and silence detection, then POSTs the transcript to the cloud chat server, then speaks the response back via the local bridge. STT uses Groq Whisper large-v3-turbo as primary (cloud, ~300ms) with Sarvam Saaras v3 as secondary fallback (good for Indian English/Hindi/Hinglish) and local faster-whisper tiny.en as final fallback. The TranscriptStabilizer class filters minor ASR variations. The FuzzyCorrector class uses rapidfuzz to correct common ASR errors (edith→edit, groq→grok, etc.). TTS has three options: Groq Orpheus (playai-tts, Fritz-PlayAI voice — currently disabled because playai-tts was removed from Groq API), Chatterbox voice clone using friend.wav (disabled on CPU because cold start takes 70+ seconds), and Piper offline TTS (en_GB-cori-high.onnx, active fallback). Piper voices are stored in voices/. The _tts_worker runs as a background thread draining a bounded queue of sentences (maxsize=20). The CrossProcessEvent class syncs TTS active state across multiple processes (widget, daemon, wake listener) using both a threading.Event and a file flag at /tmp/edith_tts_active. Barge-in detection uses webrtcvad to detect speech frames while EDITH is speaking, stopping TTS after 5 consecutive speech frames. The whisper.cpp submodule is included for local STT fallback with the ggml-small.bin model.

COGNITIVE SYSTEM — FOUR VISIONS
---------------------------------
Vision 1 (cognitive_profile.py): Maintains a growing model of Vaibhav's goals, patterns, and blind spots. The prime directive is stored in vault (encrypted) with JSON fallback. Every session logs queries to a ChromaDB collection for drift analysis. drift_score() computes the percentage of recent queries that share at least 2 words with the prime directive keywords — this is keyword overlap, not semantic drift. detect_drift() asks the LLM for a qualitative drift analysis. At session end, propose_profile_update() asks the LLM for a single new observation about Vaibhav based on the session. run_scheduled_drift_check() runs as a background task and pushes alerts if drift > 0.7. Vision 2 (self_improve.py): Fetches ArXiv paper abstracts on rotating topics (LLM agents, RAG, local AI assistants) on Tuesdays and Fridays at 10am. propose_upgrade() asks the LLM to analyze the papers and propose one concrete EDITH-specific upgrade in a structured format. The score_proposal() function scores proposals on actionability and EDITH-relevance keywords (0.0-1.0). Results are persisted in upgrade_proposals.json. High-scoring proposals are pushed to Telegram via the event bus. Vision 3 (life_os.py): simulate_decision() presents a major decision and generates 5 life simulation branches (Most Likely, Best Case, Regret-Minimised, Contrarian, Do-Nothing) with 6-12 month outlooks. weekly_briefing() generates a structured weekly report covering top priority, leverage actions, drift alerts, open loops, and one non-obvious insight. Open loops (unfinished tasks) are tracked in a ChromaDB collection with add_open_loop() and close_open_loop(). Vision 4 (council.py): run_council() runs a parallel 4-persona debate using concurrent.futures.ThreadPoolExecutor. The personas are Strategist (long-term ROI thinking), Critic (flaws and failure modes), Builder (pragmatic implementation), and Wildcard (contrarian creative thinking). Each persona has persistent memory of its prior positions stored in separate ChromaDB collections. Round 2 is adversarial cross-examination where each persona attacks the weakest argument. A synthesis LLM call generates consensus, strongest dissent, and recommended action. quick_council() is a faster 2-persona debate (Strategist vs Critic) used for lighter complex queries.


AGENT SYSTEM
------------
agent.py implements a full state machine agent: PLANNING → EXECUTING → VALIDATING → REPLANNING → DONE/FAILED. Every state transition is persisted to SQLite (agent_runs.db). plan_task() asks the LLM to break the task into max 3 numbered steps with absolute paths and no "cd" commands. get_command() converts each step into a raw bash command. compute_confidence() scores each command (0.0-1.0) based on safety signals. is_dangerous() checks against DANGEROUS_PATTERNS list and additional regex patterns for variable expansion, backtick execution, reverse shells, and remote code execution pipelines. Execution uses asyncio.create_subprocess_exec with a 30-second timeout and 1MB stdout cap. Sandboxing uses firejail with --private-tmp --net=none if available, otherwise runs as user with a warning. The REPLANNING state retries failed steps up to 2 times by asking the LLM for an alternative command. An interrupt mechanism (_STOP_AGENT threading.Event) allows stopping mid-execution. After completion, _maybe_create_skill() asks the LLM if the completed task warrants saving as a reusable skill in skills/. The compound intent system (compound_dag.py) detects "and then", "after that", "also", "next" patterns in input and splits them into sub-tasks, executing them as a DAG with a transaction log and rollback to last checkpoint. Max 15 nodes, max depth 5.

SESSION MANAGEMENT
------------------
session.py tracks every session with a UUID (8-char), device type (widget/telegram/voice/cli), start time, query list, and a context snapshot dictionary for cross-device continuity. Sessions are persisted to session_state.db via SQLite. track_query() logs every query to both the session and the cognitive profile's query log. At session end, a 5-part ritual runs: (1) propose_profile_update() generates one new profile observation from the session, (2) detect_drift() checks alignment with prime directive, (3) format_open_loops() shows unfinished tasks, (4) save_episode() saves the full session as an episodic memory, (5) ingest_text() extracts knowledge graph triples from session queries. The ritual output is a formatted ASCII art report. Cross-device session transfer is supported — transfer_session() updates the device field in the DB while preserving all session state. resume_session() loads a previous session by ID including all its queries and context snapshot.

REPO DNA ENGINE
---------------
repo_dna.py is EDITH's competitive intelligence engine. It clones GitHub repos (git clone --depth 1), scores and samples the top 10 most valuable files using a scoring system (Python files +10, README.md +5, requirements.txt +8, test files -5, lock files -10), reads up to 3000 chars per file, sanitizes PII from content, then runs two separate LLM analyses. The first analysis (run_llm_analysis) uses the full system prompt containing EDITH's own codebase as context and asks for steal_this, skip_this, watch_this, quick_wins, and summary in JSON format, with schema validation and one retry with stricter prompting on failure. The second analysis (run_strategic_analysis) uses a separate system prompt and asks for exactly 5 system-level capability gaps in a different JSON schema. Results are cached in memory_archive.db with a 7-day TTL keyed by (repo_url, commit_sha). Trend snapshots are saved per analysis run, allowing diff_analyses() to detect what new patterns appeared or disappeared between runs. compare_multi_repos() runs up to 3 repos in parallel with ThreadPoolExecutor. check_watched_repos() runs every Sunday via the background daemon, detecting new commits via git ls-remote and triggering re-analysis if the commit SHA changed. The self-audit endpoint runs repo_dna on EDITH's own codebase against CLAUDE.md guidelines.

SECURITY ARCHITECTURE
---------------------
vault.py (core/vault.py) stores all API keys in a Fernet-encrypted file (vault.enc) with a separate salt file (vault.salt). The vault password is never stored — entered at startup. On systemd deployment, VAULT_PASSWORD is set as an environment variable. Vault reads are lazily cached per key. api_auth.py implements fail-closed authentication for the FastAPI server — all endpoints require X-API-Key or Bearer token except a specific whitelist of PUBLIC_PATHS. API key comparison uses hmac.compare_digest to prevent timing attacks. Rate limiting middleware in middleware/rate_limit.py applies per-IP limits. CORS is configurable via EDITH_ALLOWED_ORIGINS, defaulting to localhost only. Shell execution in handlers/shell.py uses shlex.split() to prevent injection, and dangerous commands are blocked against DANGEROUS_PATTERNS. Docker sandbox in sandbox.py runs untrusted code with --network none and no host filesystem access. Input scope classification runs before any LLM call, and a danger keyword scan blocks known destructive patterns before dispatch. Private intents never leave the machine (forced to local Ollama). PII detection in smart_router.py forces local-only routing if emails, phone numbers, SSNs, credit cards, or Aadhaar numbers are detected in the prompt. The event bus uses async_dispatch=True with a ThreadPoolExecutor(max_workers=4) so event handler failures don't block the main thread.

DEVLOG AND CONVERSATION DNA

devlog.py maintains a development journal at devlog.md. It writes structured first-person entries when the user logs something with the format "log: <change> reason: <why> status: <status> error: <err> next: <plan>". SKILLS SYSTEM
-------------
skills_loader.py loads SKILL.md files from the skills/ directory. Each skill has YAML frontmatter with name, trigger (regex), and inject position (prefix or suffix). RAG SYSTEMS
-----------
memory/rag.py implements notes RAG using LlamaIndex with HuggingFace all-MiniLM-L6-v2 embeddings (local, no API key required). It creates a SmartCallLLM adapter class so LlamaIndex uses EDITH's smart_router for generation rather than OpenAI. Chunk size is 512 tokens with 50-token overlap. It indexes the notes/ directory for .txt, .md, .pdf, .py, .js, .json, .csv files. memory/code_rag.py is supposed to index both EDITH's own codebase and AyurStock Pro (CODE_DIRS in config). It uses AST parsing to extract Python function and class chunks individually rather than by line count. However, the index_codebase() function has a raise NotImplementedError statement in the middle of the function body — this means code RAG is completely broken and non-functional as of the current codebase state. This was left over from an Ollama removal and was never cleaned up. The query_code() function also has the same broken raise statement. So while the architecture for code RAG exists and is wired into the intent system, it does not work at all.


KNOWN BUGS AND BROKEN SYSTEMS
------------------------------
The following issues exist in the codebase as verified by code reading:

1. Code RAG (memory/code_rag.py) has `raise NotImplementedError` inside both index_codebase() and query_code(), making the entire code RAG feature non-functional. This was a leftover from an Ollama removal. The "rag" intent for code will fail silently or throw an exception.

2. Groq TTS (USE_GROQ_TTS = False in config.py) is disabled because the playai-tts model was removed from Groq's API. The config comment says "disabled until new model confirmed." So all voice output falls through to Piper local TTS.

3. Chatterbox voice clone (USE_CHATTERBOX = False) is disabled on CPU because cold start takes 70-120 seconds and times out. The config comment says "re-enable if GPU available."

4. feedback_tagger.py and the /api/feedback endpoint exist, but nothing in the live conversation path actually calls feedback_tagger. The thumbs up/down mechanism is wired to trace_logger for storage, and tuner.py reads from a feedback_signals table, but that table is never written to from the live path. The tuner's run_weekly_tune() tries to read feedback_signals and gracefully handles the failure, but produces no actual tuning effect.

5. agent.py has a reference to MODELS["chat"] in get_command() inside a try/except that swallows the error — `cmd = _llm(MODELS["chat"], prompt)` will fail because MODELS is not imported in agent.py, but the bare `cmd = _llm(prompt, intent="reason")` call on the next line works as the fallback. So agent works, just with dead code.

6. The conversation history has two parallel systems: a global `conversation_history` list AND a `HistoryManager` class AND per-source `_source_history` dict. They are not fully in sync. The HistoryManager is initialized from the loaded history but the `chat()` function primarily operates on the global `conversation_history` list and the `_source_history` dict, not the HistoryManager. The HistoryManager's auto-compaction logic is therefore unused.

7. drift_score() in cognitive_profile.py measures word overlap between recent queries and the prime directive text — this is keyword frequency counting, not actual behavioral drift detection. A query about "building EDITH" scores as aligned even if Vaibhav is procrastinating.

8. The ML intent classifier has only 60 training samples. This is insufficient for robust classification and the model likely underperforms on novel phrasings.

9. The notes/about_me.txt file contains only 5 lines of personal context. This severely limits EDITH's personalization since this is the primary static personal context file.

10. validate_vision_model() in validator.py references OLLAMA_URL and MODELS variables that are not imported — this validator will always raise NameError and be reported as a failed check.

CONVERSATION HISTORY MANAGEMENT
---------------------------------
orchestrator.py maintains conversation history across sessions by persisting to data/session_memory.jsonl (appends each message, rotates when >500 lines by keeping last 400). On startup it loads the last 20 turns. Per-source isolation: telegram, voice, and cli each have their own _source_history dict separate from the main widget conversation_history. During a session, if history exceeds 20 turns, compact_history() is called which keeps the first 2 and last 5 messages verbatim and summarizes the middle via LLM. If history exceeds 30 turns (MAX_HISTORY * 1.5), the oldest 4 turns are pulled out and summarized into a system message that prepends the history. The HistoryManager class exists and is initialized with existing history but is not used in the main chat() function path — it's dead infrastructure. shared_state.py maintains a separate 10-turn sliding window OrderedDict used by voice.py for STT context and by the widget frontend.

PROACTIVE SYSTEM
----------------
proactive.py subscribes to the event bus and sends Telegram notifications for: system alerts, health critical failures, calendar reminders, email arrivals, agent task completions/errors, self-improvement proposals, and phone notifications. A rate limiter prevents more than 10 pushes per topic per hour using a deque-based rolling window. Quiet hours (11pm to 7am IST) suppress non-critical alerts. IST is computed as UTC+5:30 without pytz (manual offset). The wire_alerts() function is idempotent and called at daemon startup and on proactive.py import. devlog.py sends an auto-generated status report to Telegram every 30 minutes covering uptime, circuit breaker status, disk usage, knowledge graph stats, and session info.

SEARCH SYSTEM
-------------
search.py implements web search with multiple provider fallback. SearXNG (self-hosted, free, localhost:8080) is the primary. Additional search APIs can be configured: Serper (83 calls/day free), Exa (33/day), Tavily (33/day), DuckDuckGo (unlimited, unofficial). The handlers/search.py chat fallback has an AI query rewriting step that converts ambiguous queries to better search phrases using smart_call. If the LLM response contains phrases like "I can search", "real-time access", "knowledge cutoff", etc., a search is automatically triggered as a second pass. Weather uses Open-Meteo (free, no API key) configured for Fatehpur, India with IST timezone.

VALIDATION AND HEALTH
---------------------
validator.py runs checks for: network reachability (TCP to 1.1.1.1:53), phone connectivity (kdeconnect-cli list-devices), calendar token validity (token.json existence and refresh_token field), disk space (minimum 1GB free), and available RAM (minimum 200MB). The validate_vision_model() function is broken — it references OLLAMA_URL and MODELS which are not imported. validate_all() runs all checks and can emit HEALTH_CRITICAL events to the event bus for each failure. format_health_report() produces a human-readable report. Health checks run every 30 minutes via the background daemon scheduler.
